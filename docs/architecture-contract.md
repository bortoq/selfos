# Self OS Architecture Contract

**Статус:** Обязательный  
**Версия:** 2.0  
**Дата:** 2026-06-22

---

## 1. Определение Self OS

Self OS — **система-посредник** (orchestration layer), которая управляет взаимодействием пользователя с внешними программами.

Self OS **не реализует** функциональность внешних программ. Он:

- Подключает внешние программы через плагины.
- Управляет потоком делегирования.
- Предоставляет пользователю единый интерфейс.
- Хранит всю историю активности в Activity Log.

Self OS **не является** набором инструментов, почтовым клиентом, планировщиком или браузером.

---

## 2. Основные Принципы

| Принцип | Описание |
|--------|---------|
| **Посредничество** | Self OS никогда не реализует функционал внешних программ самостоятельно. |
| **Единая точка входа** | Пользователь взаимодействует с программами **только** через Self OS. |
| **Activity Log как источник правды** | Все значимые события должны записываться в Activity Log. |
| **Делегирование по доверию** | Любое действие может быть делегировано только через систему доверия. |
| **Пользователь в контроле** | Пользователь всегда может отменить делегирование или потребовать review. |
| **Плагины как единственный способ интеграции** | Все внешние программы подключаются **исключительно** через плагины. |
| **Платформа** | Self OS — открытая платформа: пользователи создают и делятся плагинами и правилами. |

---

## 3. Правила Интеграции Внешних Программ

1. **Все интеграции должны быть оформлены как плагины**, наследующие `BaseSelfOSPlugin`.
2. Плагин обязан реализовывать метод `execute()`.
3. Встроенные плагины находятся в `plugins/`. Пользовательские — в `~/.selfos/plugins/`.
4. Прямой вызов API внешних сервисов **вне плагинов запрещён**.
5. Каждый плагин должен быть зарегистрирован в `PluginRegistry`.
6. Для распространения плагинов используется **Plugin Manifest** (`plugin.yaml`).

**Запрещено:**
- Создавать скрипты в `scripts/`, которые напрямую обращаются к внешним сервисам.
- Дублировать логику внешних программ внутри Self OS.

---

## 4. Правила Делегирования

1. Любое действие, которое может быть делегировано, должно иметь запись в `trust_thresholds` в `selfos.yaml`.
2. Перед выполнением делегируемого действия должна вызываться функция `can_auto(action_type)`.
3. При каждом успешном выполнении делегируемого действия должен увеличиваться счётчик доверия (`increase_trust`).
4. Все делегируемые действия должны сохраняться в Activity Log с указанием статуса (`review` / `auto`).
5. Пользователь может создавать **кастомные правила делегирования** в `~/.selfos/rules.yaml`.
6. Порядок принятия решения: Overrides → Custom Rules → Critical Actions → Trust System.

---

## 5. Архитектурные Запреты

**Запрещено:**

- Реализовывать функциональность внешних программ внутри Self OS.
- Создавать прямые интеграции без использования плагинов.
- Хранить бизнес-логику в CLI (`cli.py`).
- Создавать дублирующие функции создания событий (`create_*_event`).
- Обходить систему доверия при делегировании действий.
- Добавлять новые типы действий без обновления `trust_thresholds`.

## 6. Требования к Структуре Кода

| Компонент | Где должен находиться | Примечание |
|----------|-----------------------|----------|
| Встроенные плагины | `plugins/` | Авто-регистрация через `_auto_register()` |
| Пользовательские плагины | `~/.selfos/plugins/` | Установка через `selfos plugin install` |
| Бизнес-логика | `src/selfos/` | Включая `HookRegistry`, `DelegationEngine`, `ContextEngine` |
| Тесты | `tests/` | Обязательны для всего нового кода |
| Документация | `docs/` | Включая этот контракт |
| Маркетплейс | `docs/plugin-marketplace.yaml` | Каталог доступных плагинов |

---

## 7. Требования к Activity Log

1. Каждое значимое действие пользователя или системы **обязано** записываться в Activity Log.
2. Событие должно содержать минимум:
   - `id`
   - `timestamp`
   - `source`
   - `type`
   - `title`
   - `metadata`
3. При делегировании в `metadata` должно присутствовать поле `delegation_status` (`review` / `auto`).

## 8. Обязательные Компоненты Архитектуры

Для обеспечения соблюдения контракта в проекте **обязательно** должны присутствовать следующие компоненты:

| Компонент | Назначение | Место расположения |
|-----------|------------|--------------------|
| `EventFactory` | Единая фабрика создания событий | `src/selfos/event_factory.py` |
| `PluginRegistry` | Реестр и управление плагинами | `src/selfos/plugin_registry.py` |
| `BaseSelfOSPlugin` | Базовый интерфейс всех плагинов | `src/selfos/base_selfos_plugin.py` |
| `PluginManifest` | Манифест плагина (YAML) | `src/selfos/plugin_manifest.py` |
| `PluginMarketplace` | Каталог доступных плагинов | `src/selfos/plugin_marketplace.py` |
| `HookRegistry` | Точки расширения ядра (before/after/instead) | `src/selfos/hooks.py` |
| `DelegationEngine` | Движок делегирования + кастомные правила | `src/selfos/delegation_engine.py` |
| `DelegationRule` | Пользовательские правила делегирования | `src/selfos/delegation_rules.py` |

Все новые интеграции и события **обязаны** использовать эти компоненты.

## 9. Процесс Внесения Изменений

Любое изменение в проекте должно проходить проверку на соответствие данному контракту.

**При добавлении нового функционала необходимо ответить на вопросы:**

1. Является ли это действие **делегируемым**? Если да — добавить в `trust_thresholds`.
2. Требуется ли интеграция с внешней программой? Если да — создать плагин.
3. Нужно ли сохранять событие? Если да — использовать `EventFactory`.
4. Нарушает ли изменение принципы посредничества?
5. Может ли это быть реализовано как **пользовательский плагин** или **правило делегирования**?
6. Нужна ли **точка расширения (hook)** для этого функционала?

## 10. Расширение Платформы (Phase 4)

### Плагины
- Плагины создаются через `selfos plugin init` или `create_plugin()` SDK.
- Плагины распространяются через **манифест** (`plugin.yaml`).
- Плагины могут быть установлены из маркетплейса (`selfos plugin install <name>`)
  или из Git-репозитория (`selfos plugin install <url>`).
- Версионирование плагинов — семантическое (semver).

### Правила делегирования
- Пользовательские правила хранятся в `~/.selfos/rules.yaml`.
- Поддерживаемые типы условий: `always`, `never`, `trust_threshold`, `time_range`.
- Правила имеют приоритет (числовой), сортируются по убыванию.

### Хуки (HookRegistry)
- Плагины могут подписываться на события ядра через `on_register()`.
- Три типа хуков: `before` (изменение контекста), `after` (изменение результата),
  `instead` (замена операции).
- 12 предопределённых точек расширения (note:create, task:create, email:send, ...).

### Маркетплейс
- Каталог плагинов: `docs/plugin-marketplace.yaml`.
- `selfos plugin search` — поиск по имени/тегам/описанию.
- `selfos plugin update` — проверка и применение обновлений.
- Поля: рейтинг, количество загрузок, теги, зависимости.
